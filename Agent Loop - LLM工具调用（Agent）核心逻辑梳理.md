# LLM 工具调用（Agent）核心逻辑梳理
## 一、核心设计原则
**Stateless（无状态）**：
- 已通过 `tools write_file` 完成文件写入操作
- 核心依赖：通过**完整历史上下文**驱动后续流程，不依赖本地状态存储

---
## 二、核心执行流程
### 1. 工具调用闭环逻辑
1.  **工具执行后回传上下文**
    工具（如 `write_file`）运行完成后，通过 `sendMessage` 向 LLM 发送**全部历史上下文**，触发新一轮任务处理，用于调用第二个工具。
2.  **LLM 响应与循环触发**
    LLM 每次响应时，会根据特定场景（`tool_call` 工具调用完成/正常/异常），重新向 LLM 发送消息，驱动流程继续。
3.  **OpenAI finish_reason 分支判断**
    基于 OpenAI 接口的 `finish_reason` 字段判断终止原因：
    - 若为工具调用相关原因：再次发起工具调用流程
    - 若为正常结束：终止流程
4.  **Agent 循环控制**
    工具调用需要将控制权交还给 Agent（代码执行层），因此必须设计**循环机制**：
    - Agent 执行完工具后，重新调用 LLM
    - 每次调用都携带**全部历史上下文**，保证状态一致性

---
## 三、关键设计要点
| 要点 | 说明 |
| :--- | :--- |
| 控制权流转 | LLM 负责决策 → Agent 执行工具 → 工具执行完成 → 回传上下文给 LLM → 循环 |
| 状态维护 | 完全依赖历史上下文，Agent 本身无状态存储 |
| 循环触发条件 | 工具调用完成、异常、正常响应，以及 `finish_reason` 为工具调用时 |
| 核心依赖 | `sendMessage` 传递完整上下文、`finish_reason` 做流程分支判断 |

---
## 四、流程简化总结
```
LLM 决策 → 调用工具 → 工具执行 → sendMessage 回传全量上下文 → LLM 基于上下文继续处理
          ↑                                 ↓
          └────────── 循环（基于 finish_reason 判断是否继续） ──────────┘
```
